## 2. ブランチを使ってみる（45分）

### ブランチとは？

ここまでの作業は、すべて1本の道（`main`ブランチ）の上で進めてきました。

「ブランチ」を使うと、**本線から枝分かれして作業** できます。
枝分かれした先でどんな変更をしても、本線には影響しません。
変更がうまくいったら、あとから本線に合流（マージ）させます。

```
        main:  A --- B --- C
                             \
        add-goals:            D --- E  （ここで自由に作業）
```

### 2-1. 今いるブランチを確認する

```
git branch
```

`* main` と表示されます。`*` が今いるブランチです。

### 2-2. 新しいブランチを作って移動する

```
git switch -c add-goals
```

> **解説：**
> - `switch -c` は「新しいブランチを作って（create）、そこに切り替える（switch）」コマンドです
> - `add-goals` はブランチの名前です。「目標を追加する」という意味でつけました

確認してみましょう。

```
git branch
```

`* add-goals` と表示されていれば、新しいブランチに移動できています。

---

### コラム：`git switch` と `git checkout`

ネットの記事や書籍では、ブランチの切り替えに `git checkout` を使っている例を
たくさん見かけると思います。`git switch` と何が違うのでしょうか？

もともとGitには `git checkout` というコマンドがあり、
これ1つで「ブランチの切り替え」と「ファイルの復元」という
まったく異なる操作を担っていました。

これが混乱の原因になっていたため、Git 2.23（2019年）で役割が分離されました。

| 旧コマンド | 新コマンド | 用途 |
|-----------|-----------|------|
| `git checkout -b ブランチ名` | `git switch -c ブランチ名` | 新しいブランチを作って切り替える |
| `git checkout ブランチ名` | `git switch ブランチ名` | ブランチを切り替える |
| `git checkout -- ファイル名` | `git restore ファイル名` | ファイルの変更を元に戻す |

`checkout` も今でも使えますが、今回の研修では新しい `switch` を使っていきます。
配属先のチームが `checkout` を使っていても、やっていることは同じなので安心してください。

## ここから先の全体像：マージを2つの方法で体験する

ブランチで枝分かれした変更を、本線（`main`）に合流させることを **マージ** と呼びます。

これから、マージを **2つの異なる方法** で体験します。
やっていることは同じ「合流」ですが、**どこで操作するか** が違います。

```
【3章】自分のPC上でマージする
┌──────────────────────────────────────────────┐
│                                              │
│   自分のPC（ローカル）                         │
│   ┌──────────────────────────────────┐       │
│   │  add-goals ブランチ               │       │
│   │       ↓  git merge               │       │
│   │  main ブランチに合流              │       │
│   └──────────────────────────────────┘       │
│       ↓  git push                            │
│                                              │
│   GitHub（結果を送るだけ）                     │
│   ┌──────────────────────────────────┐       │
│   │  マージ済みのmainを受け取る        │       │
│   └──────────────────────────────────┘       │
│                                              │
└──────────────────────────────────────────────┘

【4章】GitHub上でマージする（Pull Request）
┌──────────────────────────────────────────────┐
│                                              │
│   自分のPC（ローカル）                         │
│   ┌──────────────────────────────────┐       │
│   │  add-message ブランチで作業       │       │
│   └──────────────────────────────────┘       │
│       ↓  git push（ブランチを送る）           │
│                                              │
│   GitHub（ここでマージする）                   │
│   ┌──────────────────────────────────┐       │
│   │  Pull Requestを作る               │       │
│   │       ↓  内容を確認（レビュー）     │       │
│   │       ↓  マージする               │       │
│   │  main ブランチに合流              │       │
│   └──────────────────────────────────┘       │
│       ↓  git pull（結果を取り込む）           │
│                                              │
│   自分のPC                                    │
│   ┌──────────────────────────────────┐       │
│   │  最新のmainを受け取る             │       │
│   └──────────────────────────────────┘       │
│                                              │
└──────────────────────────────────────────────┘
```

**3章（ローカルマージ）** は、自分のPC上で `git merge` コマンドを使って合流させます。
すべて自分のPCの中で完結する操作です。

**4章（Pull Request）** は、ブランチをGitHubに送ってから、GitHub上で合流させます。
実際のチーム開発では、こちらの方法が使われます。
なぜなら、マージする前にチームメンバーが内容を確認（レビュー）できるからです。

どちらも「ブランチをmainに合流させる」という同じゴールですが、
**「どこで操作するか」と「チームに確認してもらえるかどうか」** が異なります。

では、まず3章でローカルマージから体験しましょう。

```

```

```

> **解説：** 今まで `main` だけをGitHubに送っていましたが、
> ブランチもGitHubに送ることができます。

### 4-5. GitHubでPull Requestを作る

1. ブラウザでGitHubのリポジトリページを開きます
2. 黄色いバナーで「Compare & pull request」ボタンが表示されていたら、それをクリックします
   - 表示されない場合は「Pull requests」タブ → 「New pull request」をクリックします

3. 以下を入力します
   - Title: `同期へのメッセージを追加`
   - コメント欄: 自由に書いてOKです（例：「同期へのメッセージを追加しました。」）
4. 「Create pull request」をクリックします

> **Pull Requestとは？**
> 「このブランチの変更をmainに取り込んでください」というお願いのことです。
> 略して **PR** と呼びます。
>
> 実際のチーム開発では：
> - PRを作る → チームメンバーがコードを確認（レビュー） → OKならマージ
> という流れで使います。
> 今日は自分で作って、自分で確認して、自分でマージします。

### 4-6. Pull Requestの中身を確認する

PRのページで以下を確認してみましょう。

- 「Files changed」タブをクリックすると、変更内容が見られます
- 追加した行が緑色で表示されていることを確認してください
- これが「レビュー」で確認する画面です

### 4-7. マージする

1. PRのページに戻ります（「Conversation」タブ）
2. 「Merge pull request」ボタンをクリックします
3. 「Confirm merge」をクリックします

「Pull request successfully merged and closed」と表示されればOKです。

### 4-8. ローカルを最新状態にする

```

## 5. 過去に戻してやり直す（20分）

Gitのいちばん心強い機能は、**いつでも過去の状態に戻れる** ことです。

「書いてみたけど、やっぱりやめたい」「途中から方向性を変えたい」
——そんなとき、Gitなら安全にやり直せます。

ここでは、ブランチ上で作業した内容を捨てて、
過去のある時点からやり直す方法を体験します。

### 5-1. 実験用のブランチを作る

まず `main` にいることを確認します。

```
git switch main
```

実験用のブランチを作ります。

```
git switch -c experiment
```

### 5-2. わざと失敗する変更を加える

```
notepad profile.txt
```

ファイルの一番下に、以下を追記して保存して閉じます。

```
これは失敗する実験です。この文章はあとで消えます。
あえて間違った内容を書いています。
```

記録します。

```
git add profile.txt
```

```
git commit -m "実験：あとで消す内容を追加した"
```

さらにもう1つ変更を加えます。

```
notepad profile.txt
```

さらに続きに追記して保存して閉じます。

```
さらに間違った方向に進んでいます。これも消えます。
```

```
git add profile.txt
```

```
git commit -m "実験：さらに間違った方向に進んだ"
```

### 5-3. 履歴を確認する

```
git log --oneline
```

`experiment` ブランチに、2つの「失敗した」コミットが積まれていることを確認してください。

### 5-4. 「最後のコミットだけ取り消したい」— git revert

今、ブランチと履歴はこのような状態になっています。

```
main:         A --- B --- C （昨日までのコミット。安全）
                          \
experiment:                D --- E
                           │    │
                           │    └─ 「さらに間違った方向に進んだ」
                           │       ↑ これだけ取り消したい！
                           │
                           └─ 「あとで消す内容を追加した」
                              ↑ これは残したい
```

よく見ると、2つ目のコミット E（「さらに間違った方向に進んだ」）だけが問題で、
1つ目のコミット D はまだ使えるかもしれません。

こんなとき、**ブランチごと捨てるのはもったいない**ですよね。
D を残して、E だけを取り消す方法があります。

```
【revert 前】
experiment:  D --- E
             OK     NG

        ↓ git revert E

【revert 後】
experiment:  D --- E --- Revert E
             OK    NG    Eを打ち消す
                         → 結果的にDだけの状態に戻る
```

まず、取り消したいコミットのIDを確認します。

```
git log --oneline
```

一番上に表示される「実験：さらに間違った方向に進んだ」のコミットID（左側の短い英数字）をコピーしてください。

```
git revert コピーしたID
```

エディタが開いてメッセージの入力を求められることがあります。
そのまま保存して閉じてください（Git Bashでは `:wq` と入力してEnterキー）。

> **困ったら：** エディタの操作がわからなくなったら、講師を呼んでください。

結果を確認しましょう。

```
notepad profile.txt
```

**「さらに間違った方向に進んでいます」の行が消えています！**
でも「これは失敗する実験です」の行は残っています。

ファイルを閉じてください。

履歴も確認しましょう。

```
git log --oneline
```

注目してほしいのは、**コミットが消えたわけではない** ということです。
元のコミットは履歴に残ったまま、「取り消すための新しいコミット」が追加されています。

```
（新しい） Revert "実験：さらに間違った方向に進んだ"   ← 取り消しコミット
           実験：さらに間違った方向に進んだ             ← 元のコミット（残っている）
           実験：あとで消す内容を追加した
           ...
```

> **`git revert` のポイント：**
>
> - 過去のコミットを「なかったことにする」のではなく、「打ち消すコミットを新しく作る」
> - 履歴が残るので、「何を取り消したか」が記録として追える
> - チームで作業しているときも安全に使える（履歴を書き換えないため）

### 5-5. 「やっぱり全部やめよう」— ブランチごと捨てる

`revert` で1つのコミットを取り消す方法を学びましたが、
今回は実験ブランチ自体がもう不要です。ブランチごと捨てましょう。

```
git switch main
```

```
notepad profile.txt
```

**「失敗する実験」の文章がありません。** `main` は影響を受けていません。
ファイルを閉じてください。

不要になったブランチを削除します。

```
git branch -d experiment
```

> **解説：**
> - `git branch -d` は「ブランチを削除する」コマンドです
> - `main` にマージしていないブランチは `-d` では削除できません（安全装置です）
> - 今回は「本当に捨てたい」ので、大文字の `-D` を使います

エラーが出た場合は、以下を実行してください。

```
git branch -D experiment
```

> **`-d` と `-D` の違い：**
>
> | オプション | 動作 |
> |-----------|------|
> | `-d`（小文字） | マージ済みのブランチだけ削除する（安全） |
> | `-D`（大文字） | マージしていなくても強制的に削除する |
>
> `-D` は「この変更はもう要らない」と判断したときだけ使ってください。
> 削除したブランチのコミットは、通常の方法では元に戻せません。

ブランチが消えたか確認しましょう。

```
git branch
```

`* main` だけが表示されていればOKです。

### 5-6. 今の体験を振り返る

今やったことを整理します。「やり直す」方法を2つ学びました。

```
方法1：コミット単位で取り消す（git revert）
────────────────────────────────────
  コミットA  ← 残したい
  コミットB  ← これだけ取り消したい
      ↓ git revert
  コミットA  ← そのまま
  コミットB  ← 履歴には残る
  Revert B   ← Bを打ち消す新しいコミットが追加される

方法2：ブランチごと捨てる（git branch -D）
────────────────────────────────────
  main から分岐したブランチ丸ごと不要
      ↓ git switch main → git branch -D ブランチ名
  ブランチが消える。mainは無傷。
```

> **実務で「やり直したい」と思ったら：**
>
> | 状況 | 使う方法 |
> |------|---------|
> | 特定のコミットだけ取り消したい | `git revert コミットID` |
> | ブランチの作業を丸ごと捨てたい | `git switch main` → `git branch -D ブランチ名` |
>
> どちらの場合も、**こまめにコミットしていれば、やり直しの粒度を細かくできます。**
> 1つの大きなコミットにまとめてしまうと、取り消したい部分だけを戻すのが難しくなります。
> だからこそ、**こまめにコミットすること**と**ブランチを使うこと**が大切なのです。

## 7. Day 2のまとめ

### 今日新しく使ったコマンド

| コマンド | 意味 |
|---------|------|
| `git branch` | ブランチの一覧を見る |
| `git switch -c ブランチ名` | 新しいブランチを作って切り替える |
| `git switch ブランチ名` | ブランチを切り替える |
| `git merge ブランチ名` | ブランチの変更を取り込む |
| `git branch -d ブランチ名` | ブランチを削除する |
| `git revert コミットID` | 特定のコミットを取り消す |

## 付録：コンフリクト（衝突）について

> **この内容はハンズオンでは扱いません。**
> 今後の開発演習やチーム開発で実際に遭遇したときに、読み返してください。

### コンフリクトとは

同じファイルの同じ行を、2つのブランチで別々に編集した状態でマージしようとすると、
Gitは「どちらが正しいかわからない」ため、自動でマージできません。
この状態を **コンフリクト（衝突）** と呼びます。

### コンフリクトが起きる例

```
main ブランチ:      好きな食べ物：ラーメン
feature ブランチ:   好きな食べ物：寿司
```

この状態で `git merge feature` を実行すると、ファイルの中に以下のような表示が現れます。

```
<<<<<<< HEAD
好きな食べ物：ラーメン
=======
好きな食べ物：寿司
>>>>>>> feature
```

### この表示の読み方

| 記号 | 意味 |
|------|------|
| `<<<<<<< HEAD` | 今いるブランチ（main）の内容ここから |
| `=======` | 境界線 |
| `>>>>>>> feature` | マージしようとしたブランチ（feature）の内容ここまで |

### 解決の基本的な流れ

1. ファイルを開いて、`<<<<<<<`、`=======`、`>>>>>>>` の記号を探す
2. どちらの内容を残すか（または両方を組み合わせるか）を判断する
3. 記号を含めて不要な部分を削除し、正しい内容だけを残す
4. ファイルを保存する
5. `git add` → `git commit` で解決を記録する

### コンフリクトを恐れないために

- コンフリクトは「事故」ではなく「確認作業」です
- Gitが「自動で判断できないから人間に聞いている」だけです
- こまめにコミットし、こまめにマージすることで、コンフリクトの規模を小さくできます
- 困ったらチームメンバーに相談しましょう。一人で抱え込む必要はありません
